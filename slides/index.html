Problem Statement & Context

  • Current EHR Ecosystem: Existing electronic health record interfaces (e.g., MyChart) are structurally optimized for care providers and insurance payers, resulting in suboptimal patient usability.
  • Regulatory Baseline: Under current regulations (e.g., HIPAA), patients legally own their medical records and maintain the right to access them. Systems like Epic and AthenaHealth technically permit record exportation.
  • Objective: Implement an interface capable of ingesting exported raw medical data, standardizing it, and exposing it to natural language queries via large language models (LLMs).

Iteration I: Python & SQLite Prototype

  • Architecture: Native Python ingestion pipeline leveraging sqlite3 and Tcl/Tk for the GUI.
  • Generation: DB schemas and ingestion scripts generated via Google Gemini 2.5 Pro using 100+ exported XML records.
  • Functionality: Natural language input mapped to SQL queries via Gemini to retrieve and summarize historical health data.

Iteration II: Native macOS Application

  • Framework: Swift and Xcode. Codebase heavily scaffolded using GitHub Copilot, Gemini, and GPT-5 due to lack of prior Swift familiarity.
  • Core Updates: Introduced multi-backend support, accommodating both external API calls (Gemini) and local execution (Ollama).
  • Limitations: High reliance on LLM-generated code introduced verification overhead and uncertainty regarding edge-case stability.

Iteration III: Streamlit Web Application

  • Framework: Python with Streamlit for cross-device compatibility.
  • Enhancements:
    • Implementation of AES encryption for database records and conversational logs.
    • Concurrent multi-user support structures.
    • SSH tunneling capability for routing queries to private LLM instances.
  • Compliance: Explicit UI gating confirming outputs are informational and not medical advice.

Infrastructure & Deployment

  • Compute: Hosted on a minimal DigitalOcean Droplet, optimizing for low overhead while delegating compute logic to third-party LLM APIs.
  • Networking & Routing: Domain managed via Namecheap; HTTPS routing secured via Cloudflare Origin Server Certificate.
  • Access Control: Closed system utilizing SendGrid for handling invitation-based registration workflows.

Data Ingestion Interfaces: FHIR API

  • Initial Constraint: Manual XML export/import poses a significant barrier to user adoption.
  • Integration: Developed an ingestion module utilizing the FHIR API standard to interface directly with Epic Systems' MyChart infrastructures.
  • Institutional Friction: While functional in sandbox environments, direct API access is strictly gated by individual healthcare institutions (e.g., NYU Langone) due to liability and data-exfiltration risks.

LLM API Dispatch & Authentication

  • Initial Issue: Forcing end-users to provision Google Cloud projects for Gemini API access introduces excessive configurational complexity.
  • Refactoring: Abstracted the LLM backend routing to utilize OpenRouter.
  • Result: Enables a "Bring Your Own Key" (BYOK) paradigm. Reduces application-level API cost management while exposing localized performance improvements (e.g., optimal routing for Gemini-2.5-Flash).

Liability Mitigation

  • Handling sensitive personal medical information introduces significant personal liability risks.
  • Action: Established a formal Limited Liability Company (KC Explorer LLC) via LegalZoom.
  • Asset Transfer: Transferred all IP, domain registrations, and server resources to the corporate entity to strictly separate personal liability from application performance and data handling.

A Potential Clinical Deployment Protocol

  1. Pre-Encounter Intake: Patient interacts with the MyChartExplorer interface asynchronously or within the clinical waiting environment.
  2. Inference & Verification: The LLM contextualizes the patient's immediate concerns against their verified historical EHR data.
  3. Synthesis: Generates a structured pre-visit diagnostic summary report.
  4. Review: Clinician evaluates the synthesized report prior to the encounter, hypothetically reducing intake redundancy and compressing time-to-treatment.
